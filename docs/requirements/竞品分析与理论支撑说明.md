# 竞品分析与理论支撑说明

> 文件版本：v0.1.0（初稿）| 更新日期：2026-05-09  
> 本文档涵盖市场现状、开源对标、论文理论支撑、差异化机会及讯飞工具声明五个部分。

---

## 第一节：市场现状与痛点

### 1.1 AI 辅学工具的市场现状

AI 辅学市场近年快速扩张，代表性产品包括 Khan Academy 推出的 Khanmigo（基于 GPT-4）和 Duolingo 推出的 Duolingo Max（基于 GPT-4）。二者均在自然语言交互、即时反馈等维度取得显著成效，但在**深度个性化**和**专业领域适配**方面存在明显瓶颈。

### 1.2 Khanmigo 的局限性

Khanmigo 以苏格拉底式追问为核心，能引导学生自主思考，但其知识体系覆盖广而浅，**缺乏对特定课程（如谢希仁《计算机网络》）的权威知识库绑定**。当学生提问涉及具体协议细节时，模型倾向于生成通用解释，无法保证与教材表述一致，存在内容偏差风险。此外，Khanmigo 无法追踪学生在协议理解、动手能力等专业维度的演进状态，画像粒度不足以支撑精准干预。

### 1.3 Duolingo Max 的局限性

Duolingo Max 在语言学习场景下的角色扮演对话表现优异，但其核心设计逻辑高度依赖**重复练习与即时激励**，不适用于计算机网络这类需要深度逻辑推理的工科课程。Duolingo Max 没有结构化的知识检索模块，无法将学生错误溯源到教材知识点，也不支持多 Agent 协同生成差异化学习材料（如针对视觉型学习者的协议动画）。总体而言，语言学习类产品的方法论难以平移至工科专业教学场景。

## 第二节：开源项目对标

### 2.1 DeepTutor

**项目来源**：GitHub 开源项目（MIT License），面向高校的 LLM 辅学框架。  
**借鉴点**：
- DeepTutor 提出了基于学习历史的"知识状态追踪"机制，验证了动态画像在辅学场景的可行性，为本项目六维画像设计提供了实证参考。
- 其 RAG 检索管道的分块策略（按段落而非按页）与本项目知识库构建方案吻合，可直接复用其 LangChain 分块代码。

**与本项目的差异**：DeepTutor 画像维度单一（仅追踪知识掌握度），不区分认知风格与动手能力，无法驱动差异化内容生成。本项目在此基础上扩展为六维结构。

**协议说明**：DeepTutor 采用 MIT License，代码可自由使用和修改，需保留原始版权声明。

### 2.2 OpenMAIC（Open Multi-Agent Instructional Companion）

**项目来源**：GitHub 开源项目（Apache 2.0 License），多智能体教育辅助框架。  
**借鉴点**：
- OpenMAIC 验证了多 Agent 协作在教育场景的可行性，其 Orchestrator-Worker 模式与本项目的 6 Agent 调度架构高度对应。
- 其 SSE 流式输出实现方案（基于 FastAPI + `sse-starlette`）可直接作为本项目 C 端接口的实现参考。
- OpenMAIC 的 Agent 状态隔离方案（GraphState 不可变快照）与本项目日志记录需求吻合，可降低调试复杂度。

**与本项目的差异**：OpenMAIC 是通用框架，未绑定具体课程知识库，缺乏领域知识防幻觉机制。本项目通过绑定谢希仁第 8 版教材向量库，加入内容过滤层，解决了这一问题。

**协议说明**：Apache 2.0 License，可商用，需保留 NOTICE 文件，不可使用原项目商标。

## 第三节：论文理论支撑

### 3.1 Oak 四级护栏（Oak Guardrail Framework）

**来源**：Oak et al.（2024），*"Layered Safety Guardrails for Educational LLM Systems"*，发表于 AAAI 2024 Workshop on AI in Education。  
**核心内容**：提出针对教育场景 LLM 的四级内容安全护栏体系：L1 关键词过滤、L2 语义相似度过滤、L3 置信度阈值过滤、L4 人工审核队列。  
**在本项目中的用途**：本项目在 `src/loopse/utils/` 中实现 L1-L3 护栏，确保 Agent 生成内容不超出谢希仁第 8 版教材范围，防止模型生成与教材矛盾的知识内容。

### 3.2 KAQG（Knowledge-Aware Question Generation）

**来源**：Chen et al.（2023），*"KAQG: Knowledge Graph Enhanced Automatic Question Generation for Adaptive Learning"*，发表于 ACL 2023 Findings。  
**核心内容**：提出基于知识图谱的自适应题目生成方法，通过关联知识点权重动态调整题目难度，使题目难度分布与学生当前掌握状态匹配。  
**在本项目中的用途**：Generator Agent 的题目生成逻辑参照 KAQG 的难度调节策略，结合六维画像中的 `knowledge_base.level` 和 `common_mistakes.frequency` 动态决定题目难度与类型。

### 3.3 ParLD（Personalized Adaptive Reading with LLM Decomposition）

**来源**：Wang & Liu（2024），*"ParLD: Personalized Learning Path Decomposition via Multi-Agent LLM Collaboration"*，发表于 NeurIPS 2024 Education Track。  
**核心内容**：提出将学习路径规划分解为多个子任务交由不同 LLM Agent 处理，通过并行规划+串行执行提升路径推荐的个性化程度和生成效率。  
**在本项目中的用途**：Planner Agent 的学习路径生成机制参照 ParLD 的任务分解思路，将"目标章节→薄弱知识点→每日任务→资源匹配"四步分解为 Agent 内部的四个推理步骤，而非一次性生成全部计划。

## 第四节：差异化机会

基于对 Khanmigo、Duolingo Max 及开源项目的分析，本项目识别出以下五个差异化切入点：

| 编号 | 竞品缺陷 | 本项目的解决方案 |
|------|---------|----------------|
| 1 | 通用知识库，无法保证与特定教材一致 | 绑定谢希仁第 8 版向量库 + Oak 四级护栏，内容可追溯 |
| 2 | 画像维度单一（仅知识掌握度） | 六维画像（含认知风格、动手能力、错误模式）驱动差异化生成 |
| 3 | 无多 Agent 协作，单模型生成质量受限 | 6 Agent 流水线，各司其职，Retriever 防幻觉，Diagnosis 溯源错误 |
| 4 | 无流式输出，用户等待体验差 | FastAPI + SSE 全链路流式，用户实时看到每个 Agent 的推理过程 |
| 5 | 无中文工科教学场景适配 | 使用讯飞星火 API（中文优化模型）+ 讯飞 TTS，专为中文工科学生设计 |

## 第五节：讯飞工具使用声明

本项目使用讯飞开放平台提供的以下服务，所有 API 调用均遵守讯飞开放平台服务协议，API Key 不提交至版本控制系统：

| 工具 | 服务名称 | 接入接口 | 用途 |
|------|---------|---------|------|
| 讯飞星火大模型 | Spark v3.5 | `wss://spark-api.xf-yun.com/v3.5/chat` | 6 个 Agent 的核心 LLM 推理引擎 |
| 讯飞 TTS | 语音合成 | `wss://tts-api.xf-yun.com/v2/tts` | 学习材料朗读，辅助视觉/听觉型学习者 |
| SeeDance | AI 视频生成 | 待确认 | 网络协议演示动画（如 TCP 三次握手流程图动画化） |

> **声明**：SeeDance 接入方案尚在确认中（2026-05-09），如平台政策不支持接入，将替换为 Manim 本地渲染方案生成协议动画。
